# 银行场景 Skill 化编制 Basic Rule

版本：v0.9-taxonomy
日期：2026-06-19

## 1. 核心理念

Skill 化不是为了把所有场景都设计成复杂嵌套，也不是为了证明智能体可以自动完成一切。它的本质，是把金融业务中可重复、可验证、可治理、可交付的工作方法沉淀为可复用的能力资产。

我们应从客户和业务人员能理解的真实场景出发，再判断这个场景适合沉淀为一个独立 L2 Skill、若干 L2 的组合，还是暂时只适合做成 L1 原子能力或后续 L3 高级编排。

## 2. 基本定义

1. 场景：客户真实工作中的问题入口，例如授信走访前风险分析、贷后风险监测、制度合规审核。场景是业务语言，不是技术语言。
2. L2 场景 Skill：围绕一个相对完整的业务任务形成的能力单元，应有明确用户、输入、处理逻辑、输出和验收方式。当前 SA-FDE 工作的主要建设对象是 L2。
3. L1 原子 Skill：可被多个 L2 复用的基础能力，例如企业画像、知识检索、财务异常识别、公开风险线索检索、报告摘要生成。L1 不宜凭空大量设计，应在多个场景复用后逐步沉淀。
4. L3 高级编排：跨场景、跨系统、跨多个 L2/L1 的复杂任务，适合现场 FDE 或高级骨干在客户真实项目中探索，不宜在台账初版中过早固化。

## 3. 场景与 Skill 的对应关系

1. 高频、低到中等复杂度、输入输出清晰的场景，优先设计为“场景 = 一个独立 L2 Skill”。
2. 复杂但业务边界清楚的场景，可以先确定一个主 L2，再标注可能复用的 L1；不要在台账阶段过早设计多层嵌套。
3. 涉及多系统接入、跨部门协同、权限审批、交易执行、模型训练的场景，先列为 L3/专项候选，不强行塞进一个 L2。
4. 只提供中间能力、没有独立业务交付物的内容，优先观察其是否应沉淀为 L1，而不是勉强包装为 L2。

## 4. 首批试点选择原则

首批试点优先选择以下场景：

1. 业务价值明确：能提升效率、降低风险、增强客户经营或形成可展示的业务抓手。
2. 任务边界清晰：能说清楚由谁使用、输入什么、处理什么、输出什么。
3. 数据依赖可控：可以用模拟数据、脱敏数据或公开数据进行演示和验证。
4. 结果可解释可验收：输出能被业务专家审阅，能通过样例测试和人工复核。
5. 售前展示友好：能在较短时间内讲清楚“客户问题、Skill 如何工作、结果有什么价值”。

## 5. Skill 分类口径

不要把所有信息都塞进一个 `Skill分类` 字段。一个 Skill 应按多个维度同时描述，避免把业务域、工程形态、技术实现和自由标签混在一起。

| 维度 | 回答的问题 | 典型取值 |
|---|---|---|
| 场景大类 | 服务对象是谁 | 客户金融场景 / SA-AI内化场景 |
| 业务分类 | 属于哪类业务问题 | AI信贷、信贷、贷前、贷中、贷后、风控、财富管理、投研、营销、客服、合规、团队管理、投标、方案写作、技术评估 |
| 落地形态 | 如何沉淀为工程资产 | 独立 Skill、主 Skill + L1、Agent 编排、L1 候选、L3 专项候选 |
| 实现模式 | 主要靠什么机制完成 | 规则型、LLM生成型、RAG知识检索型、数据分析型、工具调用型、Agent编排型、混合型 |
| 建设状态 | 做到哪一步 | 规划、已制作、已开发 |
| 展示状态 | 页面如何展示 | catalog、skill、demo |
| 交付真实性 | 仓库里有什么真实交付物 | 目录规划、仅Demo、真实SKILL.md、真实Skill包、真实Skill包+Demo |
| 标签 | 自由检索词 | P800、日报、净值、信创等 |

实现模式说明：

1. 规则型：适合 if-then、阈值、清单校验、固定分类等规则明确的任务。
2. LLM生成型：适合文案、摘要、纪要、方案说明、话术等生成任务。
3. RAG知识检索型：适合制度、产品、FAQ、知识库、条款引用等任务。
4. 数据分析型：适合净值走势、量化指标、统计汇总、数据采集、对比分析等任务。
5. 工具调用型：适合需要调用内部 API、MCP、EOP/Dodo/Dumate 工具或脚本的任务。
6. Agent编排型：适合目标复杂、需要多角色分析、多工具协同或多 Skill 调度的任务。
7. 混合型：适合高价值金融场景，通常需要规则、LLM、数据证据、工具调用和人工校验共同完成。

旧字段 `Skill分类` 保留为历史兼容标签，后续不再作为唯一分类依据。

业务分类采用“主分类 + 标签命中”的方式使用。登记表中的 `业务分类` 建议填写最主要的业务入口，`标签` 可以补充可被筛选命中的相关入口。例如“信贷尽调调查”主分类可以填 `信贷`，标签补充 `AI信贷、风控、贷前、尽调、授信审批`；这样用户无论从 AI信贷、信贷、风控还是贷前入口筛选，都能看到该场景。

面向 SA 培训和宣贯时，不建议只把上表作为概念介绍，而应把它嵌入提交和验收流程：

1. 新增场景时先填写场景大类、业务分类、落地形态和实现模式；如果场景横跨多个业务入口，在标签中补充相关业务词。
2. 提交 Demo 时必须同时标注交付真实性，明确是仅 Demo、真实 SKILL.md、真实 Skill 包，还是真实 Skill 包 + Demo。
3. 周会展示时要用这套分类说明自己的 Skill：服务谁、解决什么业务问题、如何落地、靠什么机制运行、仓库里有什么真实文件。
4. 评审时不只看页面是否好看，还要看分类是否准确、文件是否真实、业务规则是否可复核、结果是否可解释。

## 6. 设计禁区

1. 不为了分层而分层，不为了智能体概念而制造复杂编排。
2. 不把客户私有数据、客户特有规则、未脱敏材料带入售前演示库。
3. 不把专家经验简单理解为一堆硬编码规则；规则、模型判断、证据引用和人工复核应组合使用。
4. 不在没有数据、接口、权限和验收口径时承诺自动化闭环。
5. 不把大模型包装成最终责任主体。金融场景中，大模型更适合承担信息提取、证据归纳、推理辅助、报告生成和决策解释。

## 7. 台账使用方法

1. 先用“场景清单”与同事、客户讨论真实需求，不急于争论技术实现。
2. 对每个场景先做四件事：判断 Skill 分类、是否适合作为独立 L2、是否建议首批试点、可能复用哪些 L1。
3. 对首批候选场景，再补充样例数据、业务规则、输出模板和验收样例。
4. 对现场客户项目，只回收脱敏后的场景模式、流程经验和通用规则，不回收客户私有数据。

## 8. 未来工作计划

1. 第一阶段：完成银行场景清单讨论稿，拉同事和客户共同修订场景名称、业务价值和优先级。
2. 第二阶段：选择 3-5 个首批 L2 Skill，补齐演示数据、业务逻辑、测试样例和演示页面。
3. 第三阶段：从已验证 L2 中沉淀高复用 L1，形成企业画像、知识检索、报告生成、规则匹配等共用能力目录。
4. 第四阶段：在客户现场由 FDE 围绕真实业务构建专项 Skill，通过脱敏、抽象、重构回流到售前矩阵。
5. 第五阶段：形成“售前演示库 + 客户现场方法论 + 通用能力资产”的闭环。
