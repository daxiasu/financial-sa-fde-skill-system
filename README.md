# 金融 SA-FDE Skill 体系

本工程用于沉淀金融 SA-FDE 的两条主线：

1. 售前演示 Skill 矩阵：面向销售、SA、客户和管理层展示金融行业 AI Skill 的业务价值。
2. 客户现场 FDE 工作法：面向真实客户项目，指导 FDE 如何识别场景、抽象流程、开发 Skill、验证效果并沉淀可复用资产。

## 目录结构

```text
financial-sa-fde-skill-system/
├── presales-skill-matrix/   # 售前演示 Skill 矩阵
├── field-fde-playbook/      # 客户现场 FDE 工作法
├── shared-templates/        # 通用模板
├── demo-sites/              # 可视化演示页面
└── README.md
```

## 当前 Ground Zero Skill

第一个样板 Skill：

`presales-skill-matrix/ground-zero-credit-risk-analysis/`

名称：

授信走访前穿透式风险分析 Skill

定位：

面向银行对公授信场景，模拟信贷经理在企业走访前，通过智能体平台发起风险分析，后台路由到 Skill，读取业务规则并生成风险关注等级、风险线索、走访问询清单和补充材料清单。

## Skill 分层框架

| 层级 | 定位 | 主要建设者 | 特点 | 示例 |
|---|---|---|---|---|
| L1 原子 Skill | 公共基础能力 | 技术团队 + 业务专家共建 | 通用、可复用、偏底层能力 | 企业画像抽取、财务指标解析、司法舆情检索、关联关系识别、材料完整性检查 |
| L2 场景 Skill | 员工当前业务场景沉淀 | SA / FDE / 一线业务骨干 | 面向具体业务任务，可演示、可交付、可售卖 | 授信走访前风险分析、贷后风险监测报告、客户保障缺口分析 |
| L3 高级编排 Skill | 高级自定义解决方案 | 高级骨干员工 / 现场 FDE / 方案专家 | 非标、复杂、临时、跨 Skill 调度，带有更强自主规划能力 | 小微贷专项尽调助手、普惠业务战役分析、临时监管排查任务助手 |

核心理解：

```text
L1 是能力积木
L2 是业务场景产品
L3 是专家级解决方案或智能体编排
```

L2 是 SA-FDE 转型的主要入口。大多数 SA 不一定一开始就能独立建设复杂智能体，但可以把自己熟悉的售前场景、金融业务流程、客户常见问题，沉淀成一个个可演示、可复用、可售卖的 L2 Skill。

L3 是高级实验区和复杂任务区。它可以承接非标客户需求、临时专项任务、客户现场 FDE 的创新打法。某个 L3 方案被多次验证后，可以反向拆解为新的 L2 场景 Skill，或者抽象为新的 L1 原子 Skill。

体系循环：

```text
客户现场非标任务
  ↓
L3 高级编排 Skill
  ↓ 验证有效
拆解沉淀
  ↓
L2 标准场景 Skill
  ↓ 多场景复用
抽象公共能力
  ↓
L1 原子 Skill
```

## 体系建设原则

1. 售前库只使用模拟数据、脱敏数据或公开数据。
2. 客户现场真实数据、客户私有规则、系统接口和业务秘密不得进入售前展示库。
3. 客户现场经验可以通过脱敏、抽象、重构，沉淀为通用场景模式和模拟案例。
4. 网页可以作为售前演示工具，Demo 可以做可视化包装，但凡标记为“已开发 Skill”的文件资产必须来自仓库真实文件。
5. Skill 不是单一文件，至少应有真实 `SKILL.md`；规则、模板、测试样例、演示数据和 Demo 可以作为配套资产，但不得为了展示效果凭空补目录。
6. 每个 Skill 都应能回答：适用场景、目标用户、输入、流程、规则、输出、边界、测试样例和售前话术。

## FDE Skill 分类规则

本体系不再把 `Skill分类` 当作唯一分类字段。一个 Skill 应按多个稳定维度同时描述：

| 维度 | 回答的问题 | 典型取值 |
|---|---|---|
| 场景大类 | 这个 Skill 服务谁 | 客户金融场景 / SA-AI内化场景 |
| 业务分类 | 这是哪类业务问题 | 风控、财富管理、投研、团队管理、投标、方案写作、技术评估、资料检索、合规、车险 |
| 落地形态 | 它在体系里如何沉淀 | 独立 Skill / 主 Skill + L1 / Agent 编排 / L1 候选 / L3 专项候选 |
| 实现模式 | 主要靠什么机制完成任务 | 规则型 / LLM生成型 / RAG知识检索型 / 数据分析型 / 工具调用型 / Agent编排型 / 混合型 |
| 建设状态 | 内容做到哪一步 | 规划 / 已制作 / 已开发 |
| 展示状态 | 页面如何展示 | catalog / skill / demo |
| 交付真实性 | 仓库里到底有什么真实文件 | 目录规划 / 仅Demo / 真实SKILL.md / 真实Skill包 / 真实Skill包+Demo |
| 标签 | 自由检索词 | P800、净值走势、日报、信创等 |

旧字段 `Skill分类` 保留为历史兼容标签，不再承担主分类职责。新增或维护团队 Skill 时，优先填写 `业务分类`、`落地形态`、`实现模式` 和 `交付真实性`。

面向 SA 先锋队培训时，可以直接使用：

`shared-templates/fde_sa_training_taxonomy_insert.md`

该文档把分类体系拆成培训材料可直接插入的内容，包括建议插入位置、字段说明、验收口径和行动清单。

## 银行场景 Skill 化 Basic Rule

当前银行场景台账采用 v0.9 口径：

1. 从真实业务场景出发，而不是从技术名词或 Skill 分类出发。
2. 高频、边界清晰、输入输出明确的场景，优先设计为“场景 = 一个独立 L2 Skill”。
3. 复杂场景先识别主任务和可能复用的 L1，不在台账阶段过早设计多层嵌套。
4. L1 原子能力应在多个 L2 复用后逐步沉淀，不宜凭空大量预设。
5. 跨系统、跨部门、交易执行、模型训练等复杂任务，先列为 L3/专项候选。
6. 旧的 Skill 分类口径迁移为“实现模式”维度，不再混合业务域、落地形态和标签。

完整准则文件：

`shared-templates/bank_skill_basic_rule.md`

银行场景台账：

`shared-templates/bank_skill_matrix_ledger.xlsx`

## Excel 驱动的团队 Skill 登记

团队新增 Skill 时，优先维护：

`shared-templates/team_skill_register.xlsx`

其中的 `已制作Skill登记` sheet 是网页新增内容的数据源。

推荐字段：

- `场景大类`：`客户金融场景` 或 `SA-AI内化场景`
- `业务分类`：例如 `风控`、`财富管理`、`投研`、`团队管理`、`投标`、`方案写作`
- `落地形态`：`独立 Skill` / `主 Skill + L1` / `Agent 编排` / `L1 候选` / `L3 专项候选`
- `实现模式`：`规则型` / `LLM生成型` / `RAG知识检索型` / `数据分析型` / `工具调用型` / `Agent编排型` / `混合型`
- `展示状态`：`catalog` / `skill` / `demo`
- `交付真实性`：`目录规划` / `仅Demo` / `真实SKILL.md` / `真实Skill包` / `真实Skill包+Demo`
- `来源平台`：例如 `Dumate`、`Dodo`、`通用`
- `迁移建议`：说明是否保留平台实现、迁移为平台中立 Skill、补网页 Demo
- `Skill文件路径`：有真实 `SKILL.md` 时填写
- `Demo页面路径`：有演示页面时填写
- `标签`：只放自由检索词，不要把标签当主分类

Excel 内新增 `分类口径说明` sheet，用于给同事解释每个字段的填写规则。

网页不直接读取 `.xlsx`，而是读取：

`data/team_skill_register.json`

更新流程：

```bash
python3 scripts/excel_to_team_skill_json.py shared-templates/team_skill_register.xlsx data/team_skill_register.json
```

发布时 `publish.sh` 会自动尝试执行这一步。这样 Excel 是源数据，JSON 是网页读取的数据文件。

## 真实 Skill 文件展示

首页右侧的“真实 Skill 文件”区域读取：

`data/skill_packages.json`

该文件由脚本扫描仓库中的真实文件生成：

```bash
python3 scripts/build_skill_packages.py
```

原则是：页面只展示已经存在于仓库中的 `SKILL.md`、规则文件和配套 Demo 文件；不会为了让目录看起来完整而生成虚假的 `references/`、`scripts/`、`tests/` 文件树。
